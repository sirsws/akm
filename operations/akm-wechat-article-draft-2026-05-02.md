# [AI架构] 为什么你的智能体总给通用建议：因为它还没真正认识你

副标题：从提示词到任务前用户画像诱导，AKM 试图补上智能体构建里被忽略的一步

---

很多人做 GPTs、Gems、Claude Project、OpenClaw agent，第一步都是写 instructions。

你告诉它：

- 你是谁；
- 它应该怎么回答；
- 它应该保持什么风格；
- 它应该记住哪些偏好；
- 它应该避免哪些错误。

然后你发现一个很尴尬的问题：

它仍然经常给你通用建议。

不是模型不强。不是提示词写得不够长。更根本的问题是：**你还没有系统地把“你是谁、你要什么、你不能接受什么、你如何决策”挖出来。**

智能体平台给了我们很多承载面：

- instructions；
- memory；
- profile；
- project knowledge；
- system prompt；
- `AGENTS.md`；
- `SOUL.md`；
- `USER.md`。

但它们大多只解决一个问题：**信息放在哪里。**

真正更难的问题是：**这些位置里到底该放什么？**

这就是我最近做的 AKM。

## AKM 不是记忆工具

先说清楚，AKM 不是 memory 工具。

它不解决向量检索，不解决长上下文压缩，不解决多轮对话存储，也不和 Mem0、Zep、Graphiti 这类系统抢位置。

AKM 处理的是更早的一步：

> 在智能体执行任务之前，主动发掘用户目标、约束、偏好、风险边界、资源条件和决策风格，再把这些信息整理成可注入的用户画像。

英文可以叫：

**Agent Profile Elicitation**

中文可以叫：

**智能体用户画像诱导协议**

简单说：

不是等用户把自己解释清楚。

而是让 AI 先帮用户把自己挖清楚。

## 为什么这一步重要

很多 AI 建议之所以泛，不是因为模型不会推理。

而是因为它不知道你的真实约束。

比如健身建议。

普通 AI 很容易给你：

- 每周练五天；
- 多做深蹲硬拉；
- 加有氧；
- 控制饮食；
- 保持睡眠。

这些话没有错。

但如果用户真实情况是：

- 每周只能练三次；
- 每次只有 35 分钟；
- 膝盖不适，不能高冲击跳跃；
- 家里只有哑铃、弹力带和跑步机；
- 不喜欢长时间有氧；
- 不能接受极端节食；
- 目标是减脂但不想掉太多肌肉。

那普通建议就不够用了。

真正有用的建议必须先命中这些约束。

再比如穿搭。

普通 AI 很容易说：

- 提升质感；
- 建立基础款；
- 注意色彩协调；
- 选择适合场景的搭配。

还是没错。

但真正做决策时，问题往往是：

- 你已有衣橱里有什么；
- 哪些衣服坚决不用；
- 什么场景要穿；
- 身材比例有哪些限制；
- 预算是多少；
- 是买新衣服，还是先重组已有资产；
- 是追求显瘦、正式、轻松，还是不费脑。

如果这些信息没有被建模，AI 只能给你正确但无用的废话。

## 问题不在“记不住”，而在“没有先挖出来”

现在很多智能体讨论都围绕 memory。

这当然重要。

但我越来越觉得，memory 之前还有一层更基础的问题：

**你要记什么？**

如果一开始存进去的是碎片、情绪、随口说的话，那么 memory 越强，只是越稳定地保存噪音。

真正应该先做的是主动发掘：

- 目标是否排序；
- 硬约束有哪些；
- 偏好是否稳定；
- 哪些是禁区；
- 哪些是暂时状态；
- 哪些是长期决策风格；
- 哪些信息还缺失；
- 哪些判断只是推测。

这就是 AKM 的核心。

## 普通 Agent 和 AKM Agent 的第一步不同

这件事最容易被误解成“换一段更长的提示词”。

其实不是。

真正的差别发生在第一次对话。

普通 Agent 往往是这样：

> 用户：我想减脂，帮我安排一周训练。
>
> AI：好的，我给你安排一个每周 5 天、力量训练加有氧的计划。周一练胸肩，周二跑步，周三练腿，周四休息，周五高强度间歇，周末做全身循环。

它直接开始给方案。

问题是，它还不知道用户有没有膝盖问题、每周能练几次、家里有什么器械、能不能接受长时间有氧、饮食底线是什么。

AKM Agent 的第一步应该是这样：

> 用户：我想减脂，帮我安排一周训练。
>
> AI：先不急着给计划。我需要先确认几个约束，否则方案很容易泛化。
>
> AI：你每周最多能训练几次？每次最长多久？
>
> AI：有没有伤病或必须避开的动作？
>
> AI：你现在可用的器械有哪些？
>
> AI：你不愿意接受的训练或饮食方式是什么？
>
> AI：你的优先级是快速见效，还是低摩擦、可持续？

这才是 Profile Elicitation。

它不是让 AI 记住你说过的话。

它是让 AI 在回答前先把该问的问出来。

## 我做了一个小测试

为了避免这件事停留在概念层，我用 DeepSeek API 做了一个小测试。

测试全部使用合成用户画像，不涉及真人隐私。

同一个 synthetic persona，同一个任务，分三组输入：

第一组：不给用户画像，只给任务。

第二组：给自然语言碎片，不做结构化。

第三组：给 AKM 结构化画像。

任务：

> 请给这个用户制定一周训练建议。

普通输入只有任务。

自然语言碎片是：

> 我想减脂和保持肌肉，每周大概能练三次，每次时间不多。膝盖有点不舒服，家里有哑铃、弹力带和跑步机。我不太喜欢复杂动作，也不想做很久有氧。

AKM 输入会先整理出这样的 profile：

```json
{
  "goals": ["减脂", "保持肌肉", "改善体态"],
  "hard_constraints": [
    "每周训练 3 次",
    "每次不超过 35 分钟",
    "避免高冲击跳跃"
  ],
  "available_assets": ["哑铃", "弹力带", "跑步机"],
  "soft_preferences": [
    "动作不要太复杂",
    "不喜欢长时间有氧"
  ],
  "risk_boundaries": [
    "不做极端节食",
    "不推荐大重量深蹲或跳箱"
  ],
  "decision_style": "低摩擦、可持续、优先坚持"
}
```

然后看三组输出是否真正命中这些约束。

我关心的不是哪一段文字更漂亮，而是几个很具体的问题：

- 有没有控制在每周三次；
- 有没有控制每次 35 分钟；
- 有没有避开高冲击跳跃；
- 有没有只使用现有器械；
- 有没有避开极端节食；
- 有没有解释为什么这样取舍。

这类对比，比“提示词好不好看”更重要。

这次 10 个 synthetic profiles 的结果如下：

| 输入条件 | 总分 | 约束命中 | 风险控制 | 具体性 | 可执行性 | 用户贴合 | 取舍意识 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 无画像 | 12.1 | 1.6 | 2.5 | 2.6 | 2.8 | 1.5 | 1.1 |
| 自然语言碎片 | 24.7 | 4.5 | 4.6 | 4.2 | 4.5 | 4.2 | 2.7 |
| AKM 结构化画像 | 30.0 | 5.0 | 5.0 | 5.0 | 5.0 | 5.0 | 5.0 |

这里的 30.0 不应该被理解成“真实世界满分证明”。

它只说明：在这 10 个合成场景里，AKM 组在这个评分规则下稳定高于另外两组。

更值得看的不是“满分”，而是自然语言碎片和 AKM 的差异在哪里。

再看证据计数：

| 输入条件 | 平均命中约束数 | 平均遗漏约束数 | 平均风险违规数 | 平均使用画像细节数 |
| --- | ---: | ---: | ---: | ---: |
| 无画像 | 0.8 | 3.2 | 1.1 | 2.1 |
| 自然语言碎片 | 3.7 | 0.3 | 0.0 | 4.9 |
| AKM 结构化画像 | 4.0 | 0.0 | 0.0 | 6.2 |

自然语言碎片其实已经明显改善了输出。

它能告诉 AI “我想要什么”。

但它不一定能清楚告诉 AI “我愿意放弃什么”“什么不能碰”“哪些取舍优先”。

这也是为什么 `取舍意识` 这一项差距最大：自然语言碎片是 2.7，AKM 结构化画像是 5.0。

AKM 的 `risk_boundaries` 和 `decision_style` 字段，恰好补的是这一层。

最直观的一组输出来自健身场景。

无画像时，AI 给出的是这样的通用计划：

```text
好的，由于不清楚你的具体运动基础、目标和可用器械，我为你制定一个通用、安全、可执行的一周训练计划。你可以根据自身情况调整。

核心原则：每周训练3-4天，每次30-60分钟，训练日之间至少休息一天。

一周计划示例：
- 周二：有氧训练，快走、慢跑或骑行30-40分钟。
- 周五：有氧训练，跳绳、跑步或游泳20-30分钟，可尝试间歇。
- 周六：全身综合训练，开合跳30秒+深蹲15次+俯卧撑10次+高抬腿30秒，循环3-4轮。
```

问题在于，这个回答不知道用户每周只能训练 3 次、每次不超过 35 分钟，也不知道用户膝盖不适，需要避免高冲击跳跃。

给自然语言碎片后，输出明显改善：

```text
一周三次训练建议（每次30-40分钟）

原则：复合动作优先，避免膝盖压力，有氧短时高效。

训练安排（隔天进行，如周一、三、五）：

1. 热身（5分钟）：跑步机快走或慢跑（坡度0-2，速度4-5），膝盖不适可调低速度。

2. 力量训练（20-25分钟，每个动作3组，每组10-12次，组间休息60秒）：
   - 哑铃深蹲（膝盖不超过脚尖，可减少下蹲幅度）
   - 哑铃俯身划船
   - 哑铃推举
   - 弹力带坐姿划船
   - 哑铃平板卧推

3. 有氧（10分钟）：跑步机坡度快走，或弹力带原地踏步/高抬腿（低冲击）。

备注：
- 膝盖不适时，避免跳跃、深蹲过深、跑步机跑步。
- 减脂靠饮食控制，训练只是辅助。
- 动作不复杂，可循环使用。
```

这已经比无画像好很多。

但给 AKM 结构化画像后，输出开始严格受约束：

```text
根据你的目标（减脂、保持肌肉、改善体态）和约束，以下是低摩擦、可持续的一周训练建议，每次控制在35分钟内，仅使用可调哑铃、弹力带、跑步机。

核心原则：每次训练包含热身、力量循环（3个动作，3轮）、低强度有氧收尾，避免跳跃和长时间有氧。

每周3次训练（如周一、周三、周五）

热身（5分钟）
- 跑步机快走（坡度5-8，速度4-5）
- 弹力带肩环绕、髋关节活动（各10次）

力量循环（20分钟）
- 每个动作做10-12次，组间休息45秒，完成3轮。
- 动作1：可调哑铃高脚杯深蹲
- 动作2：弹力带俯身划船
- 动作3：可调哑铃平板卧推

有氧收尾（10分钟）
- 跑步机快走（坡度10-12，速度3-4），保持心率微喘但能说话。

必要取舍说明
- 放弃长时间有氧：用快走替代，减少关节压力，更易坚持。
- 放弃跳跃动作：避免高冲击，保护膝盖。
- 放弃大重量：用中等重量高次数维持肌肉，同时促进减脂。
- 动作简单：只选3个复合动作，降低学习成本。
```

这就是 AKM 想证明的差异：

不是模型突然变聪明了。

而是用户信息从碎片变成了智能体可执行的约束。

这组测试还有边界。

它不是临床实验，不证明真实用户一定满意，也不证明健身效果。

它只说明一件很窄但很重要的事：

在合成用户画像条件下，把任务前用户信息结构化，会明显改善 AI 对约束、风险和取舍的处理。

## AKM 和普通提示词有什么区别

普通提示词通常是：

> 你是一个专业顾问，请根据我的情况给建议。

AKM 更像：

> 在给建议前，先确认我的目标、约束、偏好、资产、风险边界和决策风格；把这些信息整理成结构化 profile；以后所有建议都必须受这个 profile 约束。

区别不在语气。

区别在流程。

普通提示词让 AI 开始回答。

AKM 让 AI 先建立用户模型。

## 为什么我觉得这是智能体构建里的空白

现在很多平台都在做智能体。

ChatGPT 有 GPTs。

Gemini 有 Gems。

OpenClaw 有 `SOUL.md`、`USER.md`、`AGENTS.md` 等上下文文件。

Claude、Codex、Cursor 这类工具也都有项目级规则、记忆或配置。

这说明大家已经承认一件事：

**智能体不能只靠当前这一句话工作。**

它需要某种稳定的上下文。

但大多数平台给的是“填空位”，不是“填法”。

用户面对一个空白 instruction、空白 profile、空白 memory 时，其实很难知道该写什么。

于是最后常见结果是：

- 写一堆性格描述；
- 写几条偏好；
- 复制一段提示词模板；
- 把很多碎片塞进 system prompt；
- 用了一阵发现还是泛。

AKM 想补的就是这里。

不是再给一个神奇 prompt。

而是给一套主动发掘用户画像的方法。

## SSRN 拒了，不代表这件事没价值

我之前把 AKM 母文和两个分支稿件投给 SSRN。

审查一个月后，三篇都被拒。

拒稿信里明确说，这反映的是平台当前重点，不是对研究本身的判断。

这件事反而让我把定位看得更清楚。

AKM 不是先等平台盖章再存在的东西。

它首先应该接受 builder 的检验：

> 它能不能让智能体在给建议前更好地理解用户？
> 它能不能减少泛泛建议？
> 它能不能提高约束命中率？
> 它能不能把用户画像变成可复用资产？

如果能，这件事就有价值。

如果不能，再多论文也没用。

Builders decide, not platforms.

## 我接下来怎么验证

我不会拿真人做大规模测试。

这太隐私，也没有必要。

我的测试边界很清楚：

1. 使用合成用户画像；
2. 用同一个模型生成三组输出；
3. 一组没有 profile，一组只有自然语言碎片，一组有 AKM profile；
4. 再用独立 API 调用做 AI 盲评；
5. 评分维度包括约束命中、风险控制、具体性、可执行性、用户贴合度和取舍意识。

第一阶段已经做了 10 个 synthetic profiles，让读者看懂差异。

如果结果稳定，我会进一步完善测试设计并扩大规模，做成更完整的可复现实验。

## 最后

过去我们总觉得，让 AI 更懂用户，靠的是更长的上下文、更好的记忆、更强的模型。

这些都重要。

但我现在更关心另一个问题：

**AI 到底有没有在开始工作前，主动把用户理解清楚？**

如果没有，后面的建议再流畅，也只是碰运气。

AKM 的核心命题很简单：

> 先建模用户，再执行任务。

这不是一个记忆问题。

这是智能体工作的起点问题。

如果你也在构建智能体，可以试一个最小动作：

下一次写 instructions 之前，先不要急着规定它怎么回答。

先让 AI 用 AKM 的方式问你几个问题：

- 我的目标是什么？
- 我的硬约束是什么？
- 我的风险边界是什么？
- 我已有的资源是什么？
- 我的偏好和决策风格是什么？

把这些挖出来，再写进 profile 或 instructions。

然后看它第一次回答有没有变化。

如果有，这个仓库就是为你准备的。
